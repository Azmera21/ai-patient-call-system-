# Architecture

This system uses a FastAPI server as a Twilio-driven "patient simulator." When Twilio places an outbound call to the target line (+1-805-439-8008), Twilio requests instructions from the /twiml endpoint, which selects a scenario (one of 15 predefined patient personas covering scheduling, refills, cancellations, and edge cases like interruptions or vague requests) and generates an opening line using Groq's llama-3.3-70b-versatile model. The AI agent's spoken response is captured by Twilio's built-in speech recognition (via <Gather input="speech">) and posted to /webhook/gather, where the LLM generates the patient's next line based on rolling conversation history (capped at the last 8 exchanges to control token usage and latency). This loop continues until the conversation reaches 14 turns, at which point the bot wraps up naturally and hangs up. Call state is tracked in-memory per CallSid and cleared on call completion via the /call-status webhook.

Key design choices: Groq was chosen for LLM inference specifically for its low latency, which is critical for keeping conversational pacing natural during a live phone call — waiting several seconds for a response would break the illusion of a real caller. Twilio's native speech-to-text and Polly TTS voices were used instead of separate STT/TTS APIs to minimize latency and complexity, since Twilio already handles the audio pipeline end-to-end. Scenarios are defined as structured prompts (age, symptoms, goal) rather than fully scripted dialogue, letting the LLM improvise naturally within a bounded persona — this produced more realistic, less robotic conversations than hardcoded scripts, at the cost of slightly less predictable phrasing.

## Bug Analysis

After calls complete, nalyze_bugs.py runs as a separate offline step: it reads each transcript, sends it to Groq's LLM with a QA-focused prompt covering common failure modes (wrong hours, weekend scheduling errors, misunderstood requests, missed urgent-case escalation, unnatural flow, leaked internal/demo language), and parses the structured JSON response into a consolidated eports/bug_report.md and eports/bugs.json, categorized by severity (HIGH/MEDIUM/LOW). This keeps bug detection decoupled from the live call pipeline — the calls run in one pass, then a second offline pass analyzes them at leisure without adding latency risk to the phone conversation itself.
